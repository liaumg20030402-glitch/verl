# GRPO vs DAPO 对照实验结果分析
(DAPO 的dynamic sampling需要自己实现，在recipe文件夹)
> 模型：Qwen3.5-27B (Dense, GDN linear attention) · Megatron + vLLM
> 数据集：blzk 医疗题（train/val 完全一致）· batch_size=480 · rollout.n=8 · lr=1e-6
> 资源：GRPO = 32×H200，DAPO = 48×H200（**仅影响 wall-clock，不影响曲线形状**）
> 步数：两者都跑到 step 137（1 epoch）
> 颜色：actor/critic 面板里 **绿=DAPO，紫=GRPO**；val 面板里 **橙=DAPO，紫=GRPO**

---

## 0. 先厘清一个前提：两个脚本到底差在哪

两个脚本 **`adv_estimator` 都是 `grpo`**。也就是说所谓的「DAPO 跑」并不是换了优势估计器，而是在 GRPO 优势之上叠加了 DAPO recipe 的几个 trick。两者真正的差异如下：

| 配置项 | GRPO 脚本 | DAPO 脚本 | 作用 |
|---|---|---|---|
| `clip_ratio_low / high` | 默认对称 0.2 / 0.2 | **0.2 / 0.28（Clip-Higher）** | 放宽对低概率 token 的「上调」裁剪 → **抑制熵塌缩、鼓励探索** |
| `clip_ratio_c` | 默认 | **10.0（dual-clip）** | 限制超大负优势的更新 |
| `filter_groups.enable` | 无 | 没开 | 丢掉「全对/全错」的 group，只留有梯度信号的难题 |
| `overlong_buffer_cfg` | 无 | **enable，buffer=8192，penalty=1.0** | 对超长回答线性扣分（response 上限 16384，最后 8192 是惩罚缓冲区）|
| `loss_agg_mode` | `token-mean`(默认) | `token-mean` | token 级 loss 聚合 |
| `kl_loss_coef` | 0.0 | 0.0 | **注意：两者 `use_kl_loss=False`，所以 KL loss 实际都没启用，这个系数都是摆设** |
| `use_kl_in_reward` | False | False | 两者都没有 KL reward 惩罚 |
| `entropy_coeff` | 0 | 0 | 两者都没有显式熵正则 |

> 关键结论：**两条 run 都没有任何 KL 锚定、也没有熵正则**。熵能不能自由漂移，完全取决于 Clip-Higher / Dynamic Sampling 这些 trick。这正是后面解释「为什么 DAPO 熵会涨」的根本原因。

---

## 1. 核心问题：DAPO 的 entropy 涨得太多了吗？

**现象**（actor 面板左上 `actor/entropy`）：DAPO（绿）的策略熵在训练中持续、明显上升；GRPO（紫）则一直维持在很低、基本平的水平。

**这是正常的，而且是 DAPO 设计预期内的行为，不是 bug。** 原因有三层：

1. **Clip-Higher 就是干这个的。** DAPO 论文提出 `clip_ratio_high > clip_ratio_low`（这里 0.28 > 0.2）的初衷，就是解决标准 PPO/GRPO 的「熵塌缩（entropy collapse）」——对称裁剪会过度压制低概率探索 token，导致策略迅速变得过度确定、丧失多样性。放宽上界后，低概率 token 被允许更大幅度上调，**熵上升是它的直接、预期副作用**。所以 GRPO 熵塌到很低、DAPO 熵抬起来，恰恰说明这两个 trick 都在按设计工作。

2. **没有 KL 锚定。** 两条 run 都关掉了 KL loss 和 KL reward，策略没有被往 ref 模型拉回的力。在这种情况下，叠加 Clip-Higher 的「向上推力」，熵自然单调往上走。

3. **Dynamic Sampling 的二次效应。** `filter_groups` 把「8 个采样全对 / 全错」的 group 丢掉，只保留模型还在纠结的中等难度题。这些题本身就是模型不确定（高熵）的样本，它们在梯度里占比更高，会进一步把平均熵往上抬。

### 那「太多」到底要看什么？判断熵是否失控的标准

熵上升本身不可怕，**可怕的是「熵爆炸 + 性能崩」**。判据：

- ✅ **健康**：熵温和上升后趋于平台，**同时 reward / val 指标继续走高或持平**，rollout 文本仍然通顺。
- ⚠️ **危险**：熵单调爆炸不收敛，**伴随 reward 掉头向下、出现复读/胡言乱语/中英混杂**。

**本实验属于「健康」一侧**，证据：

| 指标 (step 137) | DAPO | GRPO | 判读 |
|---|---|---|---|
| `val-core med-blzk reward/mean@1` | **0.9513** | 0.939 | DAPO 验证集更高，且仍在上升 |
| `val-aux score/mean` | **0.9564** | 0.939 | 同上，DAPO 占优 |
| `critic/score/mean`（训练 reward）| 0.9635 | 0.9638 | 基本持平，DAPO 没有因为熵高而崩 |
| `actor/ppo_kl` | 0.0001 | 0.0002 | 都极小，策略更新幅度温和、没有暴走 |

> **结论：DAPO 熵上升是 Clip-Higher 的设计效果，目前没有失控。** 在熵持续上升的同时，验证集准确率反而是两者里最高的、且还在爬坡——这说明熵带来的探索目前是「有收益的探索」，而不是退化。**正常。**
>
> 唯一需要持续盯的：熵是「持续上升」还是「上升后趋于平台」。如果继续训下去熵一直不收敛、并开始拖累 reward，那才需要干预（见 §4）。

## 2. 另一个更值得注意的现象：DAPO 回答长度大幅缩短

这其实比熵更值得你关注，因为变化幅度非常大：

| `response_length` (step 137) | DAPO | GRPO |
|---|---|---|
| mean | **~1752** | ~4464 |
| max | ~7078 | 16384（顶到上限）|
| clip_ratio（被截断比例）| ~0（≈0）| 0.0008（仍有截断）|

`global_seqlen/mean` 也对应地从 GRPO 的 ~3.0M 掉到 DAPO 的 ~1.13M。

**原因：这是 DAPO 脚本独有的 `overlong_buffer_cfg`（超长惩罚）造成的，和熵无关。** 回答上限 16384，最后 8192 token 是线性扣分缓冲区，`penalty_factor=1.0` 相当强。模型很快学会「把答案写短、别踩进惩罚区」，于是平均长度从 ~4800 一路压到 ~1752，GRPO 因为没有这个惩罚就一直维持长 CoT、甚至经常顶到 16384 被截断。

**好坏判断**：对这个医疗任务，**短回答 + 更高的验证准确率**通常是好事（少废话、推理更紧凑），目前看是正向的。但因为长度跌幅实在很大（≈2.7×），建议：
- 去 `rollout_data_dir` 抽看几条 DAPO 的 rollout，确认答案不是被「为了避罚而强行截短/省略推理」，而是真的简洁正确。
- 如果发现推理被压得过短影响难题，可以把 `penalty_factor` 调小（如 0.5）或把 `overlong_buffer.len` 调小一点。

## 3. 其他指标速览

- **`num_turns`**：两者恒为 2（单轮任务），正常。
- **`critic/score/min`**：早期有大量 0（甚至负，来自 overlong 惩罚），后期抬升，是正常的 RL 早期高方差 + 逐步收敛。
- **`response_length/clip_ratio`**：GRPO 紫线偶尔冲到 1e-3，说明 GRPO 有少量回答被 16384 截断（`truncation='error'` 下要留意）；DAPO 几乎为 0，因为它主动写短了。
- **`timing` / `perf`**：DAPO 用 48 卡、GRPO 用 32 卡，wall-clock 不可直接比较，**这部分差异不反映算法优劣，忽略即可**。

## 4. 结论与建议

### 结论
1. **整体实验结果正常、可信。** 同模型同数据，DAPO recipe 在验证集上稳定优于纯 GRPO（0.951 vs 0.939），符合 DAPO 论文预期。
2. **DAPO entropy 上升是正常的、设计预期内的现象**——Clip-Higher + 无 KL 锚定 + Dynamic Sampling 共同作用的结果，目的就是对抗 GRPO 的熵塌缩。**「涨得多」不等于「涨得太多」**：判据是有没有伴随性能崩，而这里 val 反而最高，所以判定为健康。
3. 真正的大变化是 **DAPO 回答长度被 overlong 惩罚大幅压短**，这是配置差异导致的，目前看正向，但建议抽查 rollout 确认质量。

### 后续如果要继续训 / 调参，建议盯这几条
- **熵曲线是否趋于平台**：继续训若熵一直爆涨且 reward 掉头，再考虑加一点 KL（`use_kl_loss=True, kl_loss_coef≈1e-3`）或负的 `entropy_coeff` 把熵压回去。
- **抽查 DAPO rollout 文本**：确认短回答是「简洁正确」而非「为避罚而欠推理」。
- **公平性提醒**：两条曲线唯一不该直接比的就是时间/吞吐（卡数不同）；算法指标（entropy / reward / response_length / val）卡数无关，可以放心横向对比。

### 实验设计上的一个小提醒
两个脚本同时改了 **多个变量**（Clip-Higher、Dynamic Sampling、overlong 惩罚一起上），所以严格说这是「GRPO baseline vs 完整 DAPO recipe」的对比，**无法单独归因到某一个 trick**。如果想知道「到底是 Clip-Higher 让熵涨、还是 overlong 让长度掉」，需要做消融（一次只开一个）。当前结论「DAPO 整体更优」是成立的，但「熵涨是哪个 trick 导致」是基于机理推断（主因 Clip-Higher）而非本实验直接证明。

## 5. 常见疑问：熵不收敛 & 同配置多次跑 val 抖动

### 5.1 只训了 137 步，熵一直在涨、没有收敛，正常吗？

**正常，但要分清「没收敛」和「发散」两件事。**

- **137 步 = `total_epochs=1` 跑完，对 RL 是很短的长度。** 在这个长度上熵没有平台期是预期之内的，不代表坏掉。判据永远是「有没有拖累性能」——而 val 在 137 步还是最高且在爬坡，所以这是「还在加燃料的上升期」，不是发散。
- **但要诚实地说：这套配置里没有任何让熵收敛的机制。** 三个力的方向是：
  - `use_kl_loss=False` + `use_kl_in_reward=False` → 没有把策略拉回 ref 的力；
  - `entropy_coeff=0` → 没有熵正则；
  - `clip_ratio_high=0.28`（Clip-Higher）→ 一个持续把熵往上推的力。

  系统里**只有「推熵上升」的力、没有「拉熵回落」的力**，所以不该期待它在短训练里自己平掉。通常隐性刹车来自 Dynamic Sampling（策略变强后越来越多 group 变「全对」被过滤、优势信号变小），但这个刹车很弱。

- **结论**：1 个 epoch 跑完就停，「熵没收敛」完全正常，按 val 挑最好的 ckpt 用即可。**若要训更久**，别指望它自收敛——要么给一个回落力（见下），要么用 val early-stop。想确认是「温和上升」还是「要爆」，最直接是多跑 1~2 个 epoch 看 val 拐不拐头。

- **想给熵一个回落力（可选，二选一即可）**：
  ```bash
  # 方案 A：加一点点 KL 锚定（最常用）
  actor_rollout_ref.actor.use_kl_loss=True
  actor_rollout_ref.actor.kl_loss_coef=1e-3      # 1e-3 ~ 1e-2，从小开始
  # 方案 B：加一点熵正则（负系数压熵；verl 里 entropy_coeff>0 是鼓励探索，要压熵用很小的正则需谨慎调向）
  actor_rollout_ref.actor.entropy_coeff=-1e-3    # 量级 1e-3，先小步试
  ```

### 5.2 同样的配置，GRPO 跑两次 val 差 1~2 个点，这么随机？

**identical-config 的 RL run 之间差 1~2 个点完全正常，RL 的随机性远大于 SFT。** `data.seed=42` 只固定了**数据 shuffle 顺序**，管不住下面这些更大的随机源：

1. **Rollout 采样本身随机**：`temperature=1.0, top_p=1.0, n=8`，每次采的 token 序列不同；早期一点差异会顺着 RL 滚雪球放大（policy gradient 高方差）。
2. **vLLM 异步 rollout 不可复现**：用了 `rollout.mode=async` + continuous batching，**完成顺序 / batch 组成每次都不同** → 数值不一致 → 采样 token 不一致。vLLM 本来就不保证逐位复现。
3. **bf16 + 分布式 reduce 的非结合性**：bf16 精度低，多卡 all-reduce 累加顺序不固定，梯度逐步累积微小差异，几十步后被放大。
4. **任务接近天花板**：val 在 0.93~0.95 已接近饱和，1~2 个点在这个区间是**小噪声**，且存在很多近似等价最优解，两次 run 落到稍不同的点很自然。

**还有一个比随机性更该修的方法学坑**：

- val 用的是 `val_kwargs.n=1, do_sample=False`（贪心单样本）。对给定 ckpt 它确定，但**单条贪心解码是对「模型真实水平」的高方差估计**——val 集不大时，几道题翻面就是 1~2 个点。
- RL 的 val 曲线本身就在**逐步抖动 ±1 个点**（看 val-core 那条线是抖着上去的）。**只比 step 137 这一个点本身就不可靠。**

**让对比可信、方差更小的做法**：

- **别比单个最终步**：比 best ckpt，或最后 K 次 eval 的平均 / 滑动平均。
- **val 用多样本平均**（比单条贪心稳得多）：
  ```bash
  actor_rollout_ref.rollout.val_kwargs.n=4          # 或 8
  actor_rollout_ref.rollout.val_kwargs.do_sample=True
  actor_rollout_ref.rollout.val_kwargs.temperature=0.6
  actor_rollout_ref.rollout.val_kwargs.top_p=0.95
  # 报 avg@n（甚至 pass@n），而不是单条贪心
  ```
- **要严谨结论就跑多 seed**，报 mean±std——单 run 的 1~2 点差不能下结论。
- 真要追复现性：所有 seed 固定 + 关掉 async（`rollout.mode=sync`），能**减小但无法消除**（vLLM / bf16 注定有残余）。

**一句话**：这个「差 1~2 点」不是配错了，是 RL 固有方差 + 正好踩在近天花板 + 用单点单样本去比，三者叠加放大了观感。
