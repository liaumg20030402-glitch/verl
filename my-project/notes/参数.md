## 三层 batch 的关系

GRPO 训练一个 step 的数据流,**从粗到细有三层**。⚠️ **重要单位说明**：`train_batch_size` 和 `ppo_mini_batch_size` 在 verl 配置层面**单位都是 prompt 数**（不是 sample 数），框架内部会自动 × `rollout.n` 转换成 sample 数（见 [ray_trainer.py:1224](../../verl/trainer/ppo/ray_trainer.py#L1224) `ppo_mini_batch_size = ppo_mini_batch_size * rollout.n`）。所以验证规则是 `train_batch_size >= ppo_mini_batch_size`，**不要试图用 rollout.n 抵消**。

```
┌─────────────────────────────────────────────────────┐
│ train_batch_size = 128 (prompt 数,配置单位)        │  ← Rollout 阶段
│ × rollout.n = 8 (每 prompt 采 8 条 response)       │
│ = 1024 条 (prompt, response) 样本                  │
└─────────────────────────────────────────────────────┘
                    ↓
        reward 打分,算 advantage
                    ↓
┌─────────────────────────────────────────────────────┐
│ ppo_mini_batch_size = 32 (prompt 数,配置单位)      │  ← 训练阶段
│ verl 内部 ×rollout.n=8 → 256 条 sample / mini-batch│
│ 1024 / 256 = 4 个 mini-batch → 每 step 4 次更新    │
│ 等价地: train_batch_size / ppo_mini_batch_size     │
│        = 128 / 32 = 4 次 optimizer.step            │
└─────────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────────────┐
│ ppo_micro_batch_size_per_gpu = 1                        │  ← 前向/反向
│ ⚠️ Megatron 下单位其实是 "每个 DP 组" 不是 "每张物理 GPU"│
│ DP 组数 = n_gpus / (TP × PP × CP) = 16 / (2×1×1) = 8    │
│ 全系统一次 forward 处理 sample = 8 × 1 = 8 条           │
│ 256 / 8 = 32 次梯度累加 / mini-batch,再 optimizer.step  │
└─────────────────────────────────────────────────────────┘
```

**报错先看这条**：`train_batch_size (X) must be >= actor.ppo_mini_batch_size (Y)` —— 就是上面的约束，两个都是 prompt 数单位的直接比较。常见错配是把 ppo_mini_batch_size 当 sample 数配大了，比如 train_batch_size=512 配 ppo_mini_batch_size=1024 就会报错。改法：降 ppo_mini_batch_size 到 ≤ train_batch_size。

## 各层的具体作用

### 1. `train_batch_size`:定义"一个 RL step"的规模

**意思:每一个训练 step,从数据集拿多少个 prompt 出来做 rollout。**

- 你的值:128
- 这 128 个 prompt 会一次性送给 vLLM 做 rollout
- 每个 prompt 采 `rollout.n=8` 条 response
- 产出 128 × 8 = 1024 条训练样本

**这是"RL 数据生产"的单位**,决定每次 reward 估计的 variance、训练的探索多样性。

### 2. `ppo_mini_batch_size`:定义"一次梯度更新"的规模

**意思:配置层面以 prompt 为单位,框架内部 × rollout.n 转成 sample 数后,每次用这么多 sample 做一次 `optimizer.step()`。**

- 你的值:32（prompt 数）→ 内部 × 8 = 256 sample / mini-batch
- 更新次数:`train_batch_size / ppo_mini_batch_size = 128 / 32 = 4 次`（一个 step 内）
- 每 256 条样本算 loss、反向、更新一次权重

**这是"监督学习里我们熟悉的 batch size"**,决定每次参数更新的 noise 和 step size。

⚠️ **必须满足 `train_batch_size >= ppo_mini_batch_size`**（两个都按 prompt 数算）。verl 在 [actor.py:224](../../verl/workers/config/actor.py#L224) 强制校验,违反直接抛 ValueError。

### 3. `ppo_micro_batch_size_per_gpu`:定义"一次前向"的规模

**意思（含巨坑）:配置层面单位是 sample,但"per_gpu" 这个名字误导 ——
在 Megatron 策略下它实际是「每个 DP 组（model replica）一次前向处理几条 sample」,
不是字面的「每张物理 GPU 一条」。**

#### 为什么不是 "per GPU" 字面意义

Megatron 的并行维度里：
- **TP（Tensor Parallel）** / **PP（Pipeline Parallel）** / **CP（Context Parallel）**
  把**同一条 sample 的同一次 forward** 切到多张 GPU 上协作（all-reduce / 流水 / 序列分片）
- **DP（Data Parallel）** 才是真正复制模型、并行处理不同 sample 的维度

所以 `TP × PP × CP` 张 GPU 合起来 = 1 个 "model replica"，**它一次只能吃 `micro_batch_size_per_gpu` 条 sample**。
名字应该叫 `micro_batch_size_per_replica` 才准确。

#### 源码实锤（[verl/utils/config.py:101-104](../../verl/utils/config.py#L101)）

```python
megatron_dp = n_gpus // (model_parallel_size * context_parallel_size)
#               ↑          ↑ = TP × PP
minimal_bsz = megatron_dp * config.actor_rollout_ref.actor.ppo_micro_batch_size_per_gpu
# 注意：乘的是 DP 组数，不是 GPU 总数
```

随后 verl 校验：
```python
real_train_batch_size = data.train_batch_size * rollout.n   # 真实 sample 数
assert real_train_batch_size % minimal_bsz == 0             # 必须能整除
```

⚠️ 注意：**EP（Expert Parallel）不算在 model_parallel_size 里**。MoE 的 EP 维度在 attention/non-MoE 层是 DP 行为、在 MoE 层是 expert 切分，verl 的公式没扣 EP。

#### 代入你的多机配置算一遍

```bash
NNODES=2 → n_gpus=16
TP=2, PP=1, CP=1, EP=8
ppo_micro_batch_size_per_gpu=1
ppo_mini_batch_size=32 (prompt)
rollout.n=8
```

```
DP 组数 = n_gpus / (TP × PP × CP) = 16 / (2 × 1 × 1) = 8 个 DP 组
全系统一次 forward 处理 sample 数 = DP组数 × micro_per_gpu = 8 × 1 = 8 条

mini-batch sample 总数 = ppo_mini_batch_size × rollout.n = 32 × 8 = 256
每个 mini-batch 内梯度累积次数 = 256 / 8 = 32 次
```

即：**16 张 GPU 一次 forward 只处理 8 条 sample（8 个 DP 组各 1 条），accumulate 32 次后才 optimizer.step**。

#### FSDP/DDP 策略下退化成字面意义

[verl/utils/config.py:105-106](../../verl/utils/config.py#L105) 的兜底分支：
```python
else:
    minimal_bsz = n_gpus   # FSDP/DDP：per_gpu 字面意义下 = n_gpus
```
FSDP 下每张 GPU 自己就是一个 DP rank（TP=PP=CP=1），`per_gpu` 等于 `per_dp_replica`，名字才"对得上"。

#### 调参实操要点

1. **显存按 DP 组算**：`micro_per_gpu=1` 在 Megatron 下意味着一个 model replica 一次处理 1 条 sample，replica 内 TP/PP/CP 各卡分摊这条 sample 的显存（被切了）
2. **总吞吐按 DP 组算**：全系统一次 forward 跑 `dp × micro_per_gpu` 条。想提速优先调高 `micro_per_gpu`（前提是显存够），改 TP/PP 代价大得多
3. **divisibility 必须满足**：`train_batch_size × rollout.n` 必须能被 `dp × micro_per_gpu` 整除，否则启动报错
4. **效果无关**：这是纯显存适配参数，调它不会改变训练效果（梯度累积数学上等价于大 batch）

#### 面试金句

> verl 的 `ppo_micro_batch_size_per_gpu` 在 Megatron 策略下名字有点误导，实际是**每个 DP 组（model replica）一次前向处理的 sample 数**。看 [config.py](../../verl/utils/config.py) 里 `minimal_bsz = megatron_dp × ppo_micro_batch_size_per_gpu` 这条公式就知道，乘的是 DP 组数（`n_gpus / (TP×PP×CP)`），不是 GPU 总数。TP/PP/CP 是把一次 forward 切到多卡，不增加并行处理 sample 数；只有 DP 才真正复制模型并行不同数据。FSDP 策略下每张卡就是一个 DP rank，"per_gpu" 才退化成字面意义。

## 关键区别:监督学习 vs PPO

在普通的 SFT 里,"batch size" 只有一个值,既是数据规模也是梯度更新规模。但 PPO/GRPO 有一个**"on-policy 数据要被反复训练"**的特性:

```
拿一批 rollout 数据 → 训 N 次 → 重新 rollout
```

**这就是为什么要拆成 train_batch 和 mini_batch**:

- `train_batch=128`:每次 rollout 生成 128 prompt 的数据
- `ppo_mini_batch=32`:这批数据被切成 32 份,每份训一次(总共 32 次更新)
- 然后才重新 rollout 下一批

换句话说,**一次 rollout 的数据被反复用了 32 次更新**。这就是 PPO 相对于"纯 online RL"的数据效率来源。

### 还有第四个参数:`ppo_epochs`

虽然你没显式配(verl 默认 1),但概念上还有一个:

```
ppo_epochs = 2 意味着 1024 条样本会被整体训 2 遍
→ 总共 1024 × 2 / 32 = 64 次 optimizer.step
```

## 用 PyTorch 伪代码理解

```python
# 一个 GRPO step 的内循环
for step in range(total_steps):
    # === Rollout 阶段 ===
    prompts = dataloader.next(batch_size=128)              # train_batch_size
    responses = vllm.generate(prompts, n=8)                # rollout.n
    rewards = reward_fn(responses)                         # 1024 条
    advantages = compute_grpo_advantage(rewards)           # 组内归一化
    
    # === 训练阶段 ===
    all_samples = list(zip(prompts, responses, advantages))
    
    for epoch in range(ppo_epochs):                        # 默认 1
        shuffle(all_samples)
        
        for mini_batch in chunks(all_samples, size=32):    # ppo_mini_batch_size
            # 梯度累积(显存约束)
            for micro_batch in chunks(mini_batch, size=1*8):  # micro × n_gpus
                loss = compute_ppo_loss(micro_batch)
                loss.backward()
            
            optimizer.step()                               # 每个 mini_batch 更新一次
            optimizer.zero_grad()
```

## 几个常见困惑

### Q1:train_batch 和 mini_batch 的比值代表什么?

**正确公式**（两者都是 prompt 数单位,rollout.n 在 verl 内部自动消掉）:
```
每步 optimizer.step 次数 = train_batch_size / ppo_mini_batch_size
```

你的配置:`128 / 32 = 4 次更新`。

- **比值大**(比如 16+ 次):每批 rollout 数据反复训很多次 → 数据效率高,但可能过拟合当前 batch,策略偏离 rollout 时的策略过多(off-policy bias)
- **比值小**(比如 1-4 次):数据用完就扔 → 数据效率低,但策略始终接近 on-policy,训练稳定

**经验值**:PPO/GRPO 通常 1-8 次更新是安全区。

```bash
# 你目前 128/32 = 4 次,合理
actor_rollout_ref.actor.ppo_mini_batch_size=32

# 想更稳:降到 1-2 次更新
actor_rollout_ref.actor.ppo_mini_batch_size=64   # 128/64 = 2 次

# 想更激进:升到 8 次更新
actor_rollout_ref.actor.ppo_mini_batch_size=16   # 128/16 = 8 次
```

### Q2:为什么 mini_batch 不直接等于 train_batch?

如果 `ppo_mini_batch_size = train_batch_size = 128`,那一个 step 只更新 1 次参数。这样:
- 好处:100% on-policy,最稳定
- 坏处:数据用一次就扔,**采样极贵的 rollout 白费**(你跑 35B MoE 的 rollout 很贵!)

**折中**:拆成多个 mini_batch,用 PPO clip 约束策略不飘太远,同一批数据复用多次。

### Q3:micro_batch 怎么调?

- OOM 了 → 降 `ppo_micro_batch_size_per_gpu`
- 显存还有余 → 提高它,减少梯度累积次数,训练更快
- **只影响显存和速度,不影响效果**(梯度累积在数学上等价于大 batch)

你现在 `=1`,在 H200 141G 上**太保守**,可以试 `=2` 或 `=4`,提速 2-4 倍。

## 对应到你的实际场景,建议这样调

```bash
# 数据侧保持
data.train_batch_size=128
actor_rollout_ref.rollout.n=8

# 训练侧 —— 关键改动
```

## 一张对照表

| 参数 | 作用 | 调大的影响 | 调小的影响 |
|------|------|-----------|-----------|
| `train_batch_size` | 每步 rollout 规模 | 数据多样性↑、单步更贵 | 噪声大、训练快 |
| `rollout.n` | 每 prompt 采样数 | GRPO 组内方差估计更准 | 组内优势不可靠 |
| `ppo_mini_batch_size` | 每次更新的样本量 | 更新更稳、次数减少 | 更新频繁、可能过拟合 |
| `ppo_micro_batch_size_per_gpu` | 每 DP 组一次前向的 sample 数（Megatron 下不是字面"每 GPU"） | 显存↑、速度↑ | 显存↓、速度↓ |

好问题。verl 训练时会吐出一堆 metrics,**看懂它们才能判断训练有没有走偏**。我按重要性分组给你讲,每个指标都告诉你**"健康长什么样"** 和 **"异常长什么样,说明什么问题"**。

## 一、核心业务指标:reward / score / acc

这一组直接反映"模型到底做得好不好",是你最该盯的。

### `critic/rewards/mean`(也叫 `reward/mean`)

**一个 rollout batch 内所有样本的平均 reward。**

对你的 blzk 规则奖励来说,reward 只有 0.0 和 1.0 两个值,所以 `mean` 实际上就是"正确率"。

**健康曲线:**
- **初始值**:取决于 base model 能力。Qwen3.5-35B-A3B 这种强模型,医病历质控任务**冷启动可能就有 0.3~0.6**(模型已经能输出合格 JSON,部分结论对)
- **上升斜率**:前 50-200 step 应该能看到明显上升,之后趋于平缓
- **最终值**:0.7~0.95 之间,很少能到 1.0(数据里总有模糊边界样本)

**异常情况:**

| 现象 | 说明 |
|------|------|
| 一直是 0 | reward function 有 bug,或者模型输出格式完全对不上期望 |
| 卡在某个低值不动(比如 0.15) | 格式分太严,模型学会了某个"局部最优"(比如全答合格) |
| 先上升后崩盘 | **reward hacking** 或 entropy collapse,典型 failure mode |
| 上下剧烈震荡 | learning rate 太大 / batch 太小 / KL coef 太小 |

### `critic/score/mean`

**这个是原始打分**(reward function 返回的 `score`),在某些 reward manager 里等同于 reward,**在 dapo 里可能略有不同**。

dapo manager 会做一个特殊处理:**对所有样本 reward 都相同的 group 过滤掉**(避免 entropy 塌缩)。所以:

- `score/mean` = 所有样本的原始分数均值
- `reward/mean` = 被保留用来训练的样本的分数均值(dapo 过滤后)

如果 `reward/mean` 明显高于 `score/mean`,说明 dapo 过滤了很多"全对"或"全错"的 group。**dapo 保留比例低于 30%** 就要警觉了——说明组内 reward 多样性不够。

### `acc`(accuracy,你 reward function 返回的字段)

你的 reward 函数返回了 `"acc": bool(score > 0.5)`,这会被 dapo 聚合成 `critic/acc/mean`。

对你的场景,**acc ≈ reward**,因为是 0/1 打分。但如果你后续引入格式分(比如 0.3 格式正确 + 0.7 答案正确),acc 和 reward 就会分离——**acc 看的是最终答案是否对,reward 看的是总分**。这时 acc 比 reward 更能反映"任务完成度"。

### `judge_reason` 分布(你的自定义字段)

这是你 reward function 里返回的字段,dapo 会统计每种 reason 的占比。健康状态:

```
ok:                    60-90% (训练中期应该占大多数)
mismatch:              10-30% (答错但格式对)
json_not_found:        0-5%   (格式错误,应该随训练下降)
key_mode_not_matched:  0-5%
json_parse_failed:     <1%
```

**异常**:如果 `json_not_found` 或 `key_mode_not_matched` 训练中不下降甚至上升,说明**模型没在学格式**,要检查 reward 是不是没区分度(全零导致没梯度)。

## 二、优势函数指标:advantage

### `critic/advantages/mean` 和 `critic/advantages/std`

**Advantage 反映"这条 response 比同组里的平均水平好多少"**。GRPO 里,advantage = (reward - group_mean) / group_std,所以理论上:

- **mean 应该约等于 0**(因为是组内中心化)
- **std 应该约等于 1**(因为归一化了)

**健康状态:**
- `advantages/mean` ∈ [-0.1, 0.1](非常接近 0)
- `advantages/std` ∈ [0.7, 1.3]

**异常:**

| 现象 | 说明 |
|------|------|
| `std` 持续很小(< 0.3) | 组内 reward 高度一致——要么模型全对了,要么全错了。dapo 会过滤,但如果大量组都这样,有效训练信号很少 |
| `std` 持续很大(> 2) | reward 方差爆炸,可能 reward function 返回了异常值(NaN、inf) |
| `mean` 显著偏离 0 | 归一化有问题,检查 GRPO 实现(一般是框架 bug,不是你的问题) |

### `critic/advantages/max` 和 `critic/advantages/min`

这两个对 **detect outlier** 很有用。健康范围:

- `max` ∈ [1.5, 3.0]
- `min` ∈ [-3.0, -1.5]

如果看到 `max=50` 这种值,肯定是某个样本 reward 异常飙高——去 rollout_data_dir 里找那条样本看看。

## 三、策略更新指标:loss 和 KL

### `actor/pg_loss`(policy gradient loss)

**PPO 的策略梯度损失**。这是优化的目标,但它的数值变化**反直觉**:

- **健康状态**:负数,绝对值很小(-0.01 ~ -0.1)
- **不应期待它单调下降**!pg_loss 上下波动是正常的,因为它反映的是"当前 policy 和 rollout policy 的差异 × advantage"
- 重点看它**有没有突变成 NaN/inf**——那是训崩的信号

### `actor/kl_loss` 和 `actor/kl_coef`

KL 散度:当前 policy 和 reference model(base model)的差距。

你的配置:
```bash
actor_rollout_ref.actor.kl_loss_coef=0.01
actor_rollout_ref.actor.kl_loss_type=low_var_kl
```

**健康状态:**
- `kl_loss` 初始接近 0,训练中**缓慢上升**
- 到训练后期稳定在 **0.1 ~ 2.0** 之间
- 绝对不应该爆炸到 10+(说明 policy 跑飞了)

**异常:**

| 现象 | 原因 | 修复 |
|------|------|------|
| KL 飙到 10+ | policy 偏离 base 太远,reward hacking | 调大 `kl_loss_coef`(0.01 → 0.05) |
| KL 一直 ≈ 0 | policy 没更新,reward 没梯度 | 检查 reward 是否全零 |
| KL 震荡剧烈 | batch 太小或 lr 太大 | 增大 batch 或降 lr |

### `actor/entropy_loss` 或 `actor/entropy`

策略的熵。**衡量模型输出的多样性**。

你的配置 `entropy_coeff=0`,所以 entropy 不直接进 loss,但还是会被记录。

**健康状态:**
- **初期**高(1.5 ~ 3.0),模型输出多样
- **中期**缓慢下降(0.8 ~ 1.5),模型开始收敛
- **后期**稳定在 0.3 ~ 1.0

**异常(最危险):entropy collapse**
- 如果 entropy **快速掉到 < 0.1**,模型开始对所有 prompt 输出几乎一样的内容——**rollout 采样 8 条全部一样**,失去 RL 训练信号
- 这是 GRPO 最常见的 failure mode,**dapo reward manager 就是为了防止这个**(过滤全同 group)
- 如果遇到,把 `entropy_coeff` 设为正值(比如 0.001)加回来

## 四、采样质量指标:response length / clip ratio

### `response_length/mean`

模型输出的平均长度(token 数)。

**健康状态因任务而异:**
- 带 `<think>` 的任务:500-1500 token 正常
- 不带 thinking 的 JSON 输出:100-400 token

**异常:**

| 现象 | 说明 |
|------|------|
| length 持续下降到很短(< 50) | 模型发现"输出少可以拿分",reward hacking |
| length 持续飙到 max_response_length | 模型发现"多输出废话没惩罚",或 thinking 失控 |
| length 卡在 max_response_length 不下降 | response 被 truncate 太多,`max_response_length` 要调大 |

### `response_length/clip_ratio`

**超过 `max_response_length=2048` 被截断的比例**。你的 truncation 设置是 `error`,所以如果超了会直接报错,这个指标通常是 0。但如果你改成了 `truncate`,就要关注这个:

- 健康:< 5%
- 警告:5-20%(信号被丢,reward 不准)
- 危险:> 20%(`max_response_length` 必须加大)

### `actor/ratio/mean` 和 `actor/ratio/max`(PPO ratio)

PPO 的重要性采样比率:`new_policy / old_policy`。

**健康状态:**
- `ratio/mean` ≈ 1.0(略有偏差)
- `ratio/max` < 10(偶尔的 outlier)

**异常:**
- `ratio/max` 持续 > 100:某些样本的概率变化极大,通常是 ppo_epochs 或 ppo_mini_batch 过多导致策略飘太远
- 修复:降低 `ppo_mini_batch_size / train_batch_size` 的比值

### `actor/pg_clipfrac`(被 PPO clip 裁剪的样本比例)

**健康范围**:10% ~ 30%

- 太低(< 5%):clip 机制几乎没生效,相当于纯 policy gradient,可能训练不稳
- 太高(> 50%):太多样本被 clip,实际更新的梯度信号很弱

## 先对比:SFT 训练 vs RL 训练

### SFT(监督学习):确实只有一种前向

```
batch (input, label)
    ↓ 前向传播(forward)
    ↓ 计算 loss = CrossEntropy(logits, label)
    ↓ 反向传播(backward)
    ↓ optimizer.step()
```

SFT 里只有一次"前向",label 是人工给的,loss 就是预测 vs 真值的差距。**简单、直接**。

### RL(GRPO/PPO):有多个前向,而且分工不同

```
prompt batch
    ↓ 【前向 #1: Rollout 生成】—— vLLM 里多次自回归前向
    ↓ 得到 responses
    ↓ 【Reward 阶段】—— 给 responses 打分(不是 loss)
    ↓ 得到 rewards → 算 advantages
    ↓ 【前向 #2: 计算 log_prob】—— actor 模型对已生成的 response 做一次前向
    ↓ 【前向 #3: 计算 ref log_prob】—— ref 模型做一次前向(算 KL)
    ↓ 【计算 loss】—— 用 log_prob、advantage、ref_log_prob 组合成 PPO loss
    ↓ 反向传播
    ↓ optimizer.step()
```

## 一、什么是 Rollout

**Rollout ≠ 单次前向传播。Rollout 是"让模型生成一整条 response"的过程。**

### Rollout 内部发生了什么

假设 prompt 是 "1+1=",模型要生成 "2<eos>",一共 2 个 token。**生成每个 token 都要一次前向传播**:

```
t=0: 输入 "1+1="           → 前向 → 输出 token 概率分布 → 采样出 "2"
t=1: 输入 "1+1=2"          → 前向 → 输出 token 概率分布 → 采样出 "<eos>"
                                                       → 停止
```

**所以生成一条 2 个 token 的 response = 做 2 次前向传播**。

你的配置:`max_response_length=2048`,意味着极端情况下一条 response 要做 **2048 次自回归前向**。每次前向生成 1 个 token(vLLM 有 paged attention、KV cache、batching 等优化,但逻辑上就是一次次前向)。

再乘以你的 batch:
- `train_batch_size=128` 个 prompt
- `rollout.n=8` 条 response/prompt
- **1024 条 response × 平均 500 token = 50 万次前向传播**

**这就是为什么 rollout 阶段这么慢、这么贵**。vLLM 存在的意义就是让这一步高效。

### Rollout 的特点

1. **不计算梯度**(`torch.no_grad()`),也不算 loss
2. **纯推理**,目的只是"生成数据"
3. **用 vLLM 或类似推理引擎**,因为要极致优化 KV cache、batching
4. 生成完就把 responses 交给下游处理

### Rollout 和"计算 log_prob 的前向"不是同一个

注意一个微妙区别:
- **Rollout 时的前向**:每次只算下一个 token 的概率,采样出来,丢掉其他 logits
- **训练时的前向(算 log_prob)**:对已经生成好的完整 response,一次性前向,拿到**每个位置的 log_prob**,用来算 loss

为什么要再前向一次?因为 rollout 时用的可能是 vLLM 的引擎(独立进程、BF16 推理),而训练时的 actor model 权重**正在被更新**,两者是分离的。所以训练时要用当前 actor 的权重**重新前向一次**算准确的 log_prob。

## 二、什么是 Reward 和 Advantage

**这俩都不是 loss,而是"监督信号"。**

### Reward

Reward 是**外部给的分数**,告诉模型"这条 response 好不好"。来源可以是:
- 规则函数(你的 blzk:JSON 格式对 + 结论对 = 1,否则 = 0)
- 奖励模型(用另一个 LLM 打分)
- 人工标注

**Reward 不可微**,它是一个黑盒打分函数——你的 `compute_score_blzk_rule` 里又是正则又是 JSON 解析,根本无法反向传播。**Reward 不参与梯度计算**。

### Advantage

Advantage = reward 的**相对化**表示。

在 GRPO 里:
```
advantage_i = (reward_i - group_mean) / group_std
```

含义:"这条 response 在它所属的 group(同一个 prompt 的 8 条 response)里,比平均水平好多少个标准差"。

**为什么不用 reward 直接算 loss?**

- Reward 的绝对值可能系统性偏高或偏低(比如任务很难,reward 普遍低)
- 用 advantage 消除这种偏差,让训练更稳定
- Advantage > 0 的样本:要"鼓励"这种行为
- Advantage < 0 的样本:要"抑制"这种行为

### Reward 和 Advantage 都只是"数字标签"

它们的作用类似 SFT 里的 label —— **告诉模型"这条数据的梯度方向应该是正还是负,强度多大"**。但本身不是 loss。

## 三、真正的 Loss:PPO Policy Gradient Loss

**Loss 是根据 log_prob 和 advantage 组合出来的。**

### 简化版的 PPO loss

```python
# 对于每条 (prompt, response, advantage) 样本
log_prob_new = actor_model(prompt, response)      # 当前 policy 的 log_prob
log_prob_old = rollout_log_prob                   # rollout 时 policy 的 log_prob(已保存)

ratio = exp(log_prob_new - log_prob_old)          # 重要性采样比率

# PPO clip loss
loss_unclipped = -ratio * advantage
loss_clipped   = -clip(ratio, 1-ε, 1+ε) * advantage
pg_loss = max(loss_unclipped, loss_clipped).mean()

# KL 正则
log_prob_ref = ref_model(prompt, response)
kl = log_prob_new - log_prob_ref
kl_loss = kl.mean()

# 总 loss
total_loss = pg_loss + kl_coef * kl_loss
total_loss.backward()
```

**关键点:**

1. **loss 是 `log_prob × advantage` 的组合**——log_prob 可微,advantage 是常数
2. 梯度通过 `log_prob_new` 回传到 actor 参数
3. Advantage 只是个"乘数",告诉梯度"往哪个方向走,走多大步"

### 用 SFT 对比类比

| | SFT | PPO/GRPO |
|---|-----|-----------|
| 监督信号 | label(目标 token) | advantage(标量) |
| 模型输出 | logits 上每个 token 的概率 | log_prob(已生成 token 的概率) |
| Loss | -log P(label)(交叉熵) | -log P(response) × advantage |
| 信号来源 | 人工标注的正确答案 | reward function 打分 + 组内归一化 |

可以粗略地看成:**PPO 是"带权重的 SFT"**——advantage 是每条样本的权重,rollout 生成的 response 是"伪 label",按 advantage 的正负和大小来决定学习方向和强度。

帮你把整个链路再串一遍,用你的配置数字:

```
┌─ 一个 Training Step ─────────────────────────────────────┐
│                                                           │

│                                                           │

│                                                           │

│   - Actor model 前向一次 → log_prob_new                  │

│   - 【这里的前向是一次性整句过,不是自回归】            │
│                                                           │

│                                                           │

```

## 五、一个常见的误解

> "既然 reward 不可微,那梯度怎么回传到 model 上?"

**关键 trick:梯度不从 reward 回传,而是从 `log_prob_new` 回传。**

```python
loss = -log_prob_new × advantage   # advantage 是常数(detach 过)
```

反向传播时:
- `advantage` 被视为常数(`.detach()` 处理过)
- `log_prob_new` 是可微的(actor 前向算出来的)
- 梯度通过 `log_prob_new` 传到 actor 的参数

这就像**"用当前模型生成的 token 当伪标签,但给每条样本加了 advantage 的权重"**——如果 advantage > 0(这条比平均好),就鼓励模型更愿意输出这种 token;如果 < 0,就抑制。

## 简洁回答你的原问题

| 你的理解 | 实际是 |
|---------|--------|
| LLM 的 rollout 就是前向传播 | ❌ Rollout 是"用模型生成完整 response"的过程,**包含几百到几千次自回归前向**,而且和训练时的前向是分开的(vLLM vs actor) |
| 计算 loss 就是计算 reward 和 advantage | ❌ Reward 和 advantage 是**监督信号(标签)**,不是 loss。真正的 loss 是 `-log_prob × advantage`(PPO clip + KL 正则),计算过程只有 **log_prob** 参与梯度,reward 和 advantage 只是**数值权重** |

# ppo_max_token_len_per_gpu 是什么？

短答：**当 `use_dynamic_bsz=True` 时，每张 GPU 每个 micro-batch 可以打包的最大 token 数上限。** 它是 dynamic batching 的"放料尺"。

## 静态 batch vs 动态 batch

### 静态（`use_dynamic_bsz=False`）—— 你目前的配置

```bash
actor_rollout_ref.actor.use_dynamic_bsz=False
actor_rollout_ref.actor.ppo_micro_batch_size_per_gpu=1
```

每张 GPU 每次前向**固定 1 条样本**，不管这条是 100 token 还是 16384 token。

问题：**填不满显存 / 长样本撑爆显存**。比如 prompt+response = 2000 token 的样本，H200 141G 完全可以一次塞 4 条；但同 batch 里如果有一条 16000 token 的，就只能 1 条。固定 batch 取的是"最坏情况"的安全值，平均 GPU 利用率低。

### 动态（`use_dynamic_bsz=True`）

```bash
actor_rollout_ref.actor.use_dynamic_bsz=True
actor_rollout_ref.actor.ppo_max_token_len_per_gpu=16384   # 每卡最多塞 16384 个 token
```

verl 会把 mini-batch 里的样本**按真实长度装箱**：每张卡上能凑多少条就凑多少条，**只要总 token 数 ≤ ppo_max_token_len_per_gpu**。短样本一卡塞 8 条，长样本一卡塞 1 条，**显存利用接近上限**。

源码位置（[verl/workers/actor/megatron_actor.py:225-233](../../verl/verl/workers/actor/megatron_actor.py#L225)）：
```python
use_dynamic_bsz = data.meta_info.get("use_dynamic_bsz", False)
if use_dynamic_bsz:
    assert max_token_len is not None, "max_token_len must be set when use_dynamic_bsz is True"
```

## 几个相关但不同的参数

你的脚本里其实有**三个**长得很像的 `*_max_token_len_per_gpu`：

| 参数 | 用在哪个阶段 | 用途 |
|---|---|---|
| `actor_rollout_ref.actor.ppo_max_token_len_per_gpu` | actor **训练**前向+反向 | 训练时打包上限 |
| `actor_rollout_ref.rollout.log_prob_max_token_len_per_gpu` | rollout 后**重算 old_log_prob**的前向 | 见下文 |
| `actor_rollout_ref.ref.log_prob_max_token_len_per_gpu` | reference 模型前向（算 KL）| 见下文 |

注意 `rollout.log_prob_*` 这个名字有迷惑性 —— 它**不是 vLLM 推理用的**，而是 **Megatron 在 rollout 完成后、用训练引擎重新 forward 一遍 (prompt, response) 拿 old_log_prob 时的批大小**（HybridEngine 必须 recompute old_log_prob 来保证 PPO ratio 数值自洽，详见 [reward解答.md](./reward解答.md)）。

**为什么训练和 forward-only 要分开？**
- 训练前向需要保留激活做反向，显存压力大 → `ppo_max_token_len_per_gpu` 通常设小
- forward-only 只算 logprob，无反向 → `log_prob_max_token_len_per_gpu` 可以设大（一般是训练值的 2-3 倍）

参考 deepseek671b 的官方示例：
```bash
actor_ppo_max_token_len=$(((max_prompt_length + max_response_length) * 1))   # 训练用
infer_ppo_max_token_len=$(((max_prompt_length + max_response_length) * 3))   # forward-only 用
```

## 你的脚本现状的判断

```bash
data.max_prompt_length=8192
data.max_response_length=16384
actor_rollout_ref.actor.ppo_max_token_len_per_gpu=4096   # ← 太小了
actor_rollout_ref.actor.use_dynamic_bsz=False            # ← 没启用 dynamic
```

`use_dynamic_bsz=False` 时，**`ppo_max_token_len_per_gpu` 这一行根本没被读**，是死代码。但 8192 + 16384 = 24576 token/条，按你 micro_bs_per_gpu=1，**单卡每个 micro-batch 就可能要塞 24576 token**，比 ppo_max_token_len_per_gpu=4096 大 6 倍。如果哪天打开 dynamic_bsz 而忘了改这个值，会直接 OOM 或 dynamic batching 失效。

**改进建议（启用 dynamic）**：
```bash
actor_rollout_ref.actor.use_dynamic_bsz=True
actor_rollout_ref.actor.ppo_max_token_len_per_gpu=$((1 * (8192 + 16384)))   # 1 倍序列长度
actor_rollout_ref.rollout.log_prob_use_dynamic_bsz=True
actor_rollout_ref.rollout.log_prob_max_token_len_per_gpu=$((3 * (8192 + 16384)))  # 3 倍
actor_rollout_ref.ref.log_prob_use_dynamic_bsz=True
actor_rollout_ref.ref.log_prob_max_token_len_per_gpu=$((3 * (8192 + 16384)))
```

注意 [verl/workers/engine_workers.py:550](../../verl/verl/workers/engine_workers.py#L550) 有断言：
```python
assert self.config.actor.use_dynamic_bsz == self.config.rollout.log_prob_use_dynamic_bsz
```
**actor 和 rollout 的 dynamic_bsz 开关必须一致**，要么都 True 要么都 False，否则启动失败。

## 面试要点

- 静态 batch 是"按条数"，动态 batch 是"按 token 数"；后者把变长样本装箱，显存利用率高
- `ppo_max_token_len_per_gpu` 是 dynamic 的"装箱容量"上限；`use_dynamic_bsz=False` 时它无效
- 训练阶段和 forward-only 阶段的容量上限可以**不一样**（forward-only 没反向，可以更大）
- 调法经验：先开 dynamic，把 max_token_len 设成"序列总长 × 1"作为起点，OOM 就降，不 OOM 就升

# use_kl_loss = True vs False 的区别

短答：**`use_kl_loss` 控制"KL 散度作为 loss 项直接加进 policy_loss"**。它和 `algorithm.use_kl_in_reward` 是**两种不同位置施加 KL 约束**的方式。

## 两种 KL 施加位置

PPO/GRPO 防止策略偏离 reference model 太远，有两种位置可以塞 KL 惩罚：

### 方式 A：`use_kl_loss=True` —— KL 加在 loss 里（推荐，GRPO 默认）

```python
# verl/workers/actor/megatron_actor.py:562-570
if self.config.use_kl_loss:
    ref_log_prob = data["ref_log_prob"]
    kld = kl_penalty(logprob=log_prob, ref_logprob=ref_log_prob, kl_penalty=self.config.kl_loss_type)
    kl_loss = agg_loss(loss_mat=kld, loss_mask=response_mask, ...)
    policy_loss = policy_loss + kl_loss * self.config.kl_loss_coef    # ← 直接加
```

数学形式：
```
total_loss = pg_loss + kl_loss_coef * KL(π_θ || π_ref)
```

KL 作为**正则项**直接进入 loss，梯度通过 KL 项回传。

### 方式 B：`algorithm.use_kl_in_reward=True` —— KL 从 reward 里扣

```python
# 简化逻辑
shaped_reward = reward - kl_coef * (log_prob_θ - log_prob_ref)
advantage = compute_advantage(shaped_reward)   # 用扣过 KL 的 reward 算 advantage
loss = -log_prob_θ * advantage                 # 标准 PPO loss，不另外加 KL 项
```

KL 被"贴到 reward 上"，**通过 advantage 间接影响梯度**。

### 方式 C：两个都开

verl 不禁止，但会打 warning（[verl/utils/config.py:169](../../verl/verl/utils/config.py#L169)）：
```python
if config.algorithm.use_kl_in_reward and config.actor_rollout_ref.actor.use_kl_loss:
    print("NOTICE: You have both enabled in-reward kl and kl loss.")
```
等于 KL 收两次税，一般没必要。

## 二者的本质区别

| | use_kl_loss=True（loss 里） | use_kl_in_reward=True（reward 里）|
|---|---|---|
| 梯度路径 | KL 项直接产生梯度 | KL 通过 advantage 间接影响 |
| 信号强度 | 每条样本都受约束 | 受 advantage 缩放，advantage=0 的样本无约束 |
| 直观性 | 像传统 SFT 加正则，好理解 | 改造 reward，不直观但更"端到端" |
| 调参 | `kl_loss_coef` 直接对应正则强度 | `kl_coef` 进 reward，量级和 reward 耦合 |
| GRPO 论文推荐 | ✅ 是的 | ❌ 一般不用 |
| 经典 PPO（OpenAI 风格） | 不一定用 | ✅ 经典做法 |

## use_kl_loss=False 的情况

如果 `use_kl_loss=False` 同时 `use_kl_in_reward=False`，**KL 完全不约束**。这时：
- ✅ 训练初期可能学得快（无约束）
- ❌ 容易 reward hacking、entropy collapse、与 base 拉开太大导致泛化崩
- 需要靠 PPO 的 ratio clip 单兵作战，clip 一旦不够紧（`clip_ratio_high` 太大）就翻车

GRPO 论文里明确建议用 `use_kl_loss=True`，并且 `kl_loss_type=low_var_kl`（k3 estimator，方差比朴素 KL 小）。

## 你的配置评估

```bash
actor_rollout_ref.actor.use_kl_loss=True
actor_rollout_ref.actor.kl_loss_coef=0.01
actor_rollout_ref.actor.kl_loss_type=low_var_kl
algorithm.use_kl_in_reward=False
```

✅ **完全符合 GRPO 推荐配置**：
- KL 通过 loss 项约束（直接、可控）
- 不重复在 reward 里扣
- 用低方差估计器
- 系数 0.01 是个温和的值（DAPO 论文里 0.001 ~ 0.01 都常见）

## 怎么调

观察训练时的 `actor/kl_loss` 这个 metric：

| `kl_loss` 走势 | 说明 | 怎么改 |
|---|---|---|
| 缓慢上升到 0.5~2.0 后稳定 | 健康 | 不动 |
| 飙到 10+ 不收敛 | KL 约束太弱，policy 跑飞 | `kl_loss_coef` 提高（0.01 → 0.05）|
| 一直 ≈ 0 | 约束过强，policy 学不动 / 没有 reward 梯度 | 降低 `kl_loss_coef`，或先查 reward 是否全 0 |
| 抖动剧烈 | batch 太小 / lr 太大 | 增大 batch 或降 lr |

## 面试要点

- KL 约束有"加 loss"和"扣 reward"两种位置，**等价但不一样**
- GRPO 默认用 `use_kl_loss=True`（DeepSeek-Math、DAPO 都这么配）
- KL 散度计算方式（`kl_loss_type`）也有讲究：`low_var_kl` 即 k3 estimator，比朴素 KL 方差小，是 [John Schulman 推荐做法](http://joschu.net/blog/kl-approx.html)
- 全关 KL（两个都 False）= 纯靠 PPO clip，激进但不稳，除非搭配很严的 clip_ratio

# 为什么 Qwen3.5 必须 use_remove_padding=False + use_dynamic_bsz=False？

短答：**Qwen3.5 用的 Gated Delta Net (GDN) 线性注意力当前不支持 packed sequence (THD) 格式**，必须强制走传统 padded (BSHD) 计算路径，所以这三个参数被锁死成 False。这个限制写在官方示例 [run_qwen3_5-35b-megatron.sh](../../verl/examples/grpo_trainer/run_qwen3_5-35b-megatron.sh) 头部的注释里：

```bash
# Qwen3.5 architecture notes:
#   Qwen3.5 uses Gated Delta Net (GDN) linear attention which currently does
#   NOT support packed sequences (THD format) in Megatron-LM. Therefore:
#     - model.use_remove_padding=False           (forces bshd compute format)
#     - actor.megatron.use_remove_padding=False  (forces bshd compute format)
#     - actor.use_dynamic_bsz=False              (required for bshd mode)
#
#   Once Megatron-LM adds THD support for Qwen3.5 GDN, use_remove_padding
#   can be set to True for better performance.
```

要懂为什么是这三个参数，要先搞清楚 **THD vs BSHD** 是什么。

## 一、THD vs BSHD：两种张量布局

### BSHD（传统格式，padded）

```
Batch × Seq × Hidden × Dim
```

batch 里每条样本**长度不同**，但都 pad 到 batch 最大长度，shape 整齐：

```
样本1: [t1 t2 t3 PAD PAD]
样本2: [t1 t2 t3 t4  t5 ]
样本3: [t1 PAD PAD PAD PAD]
        ↓
Tensor: [3, 5, hidden, dim]  ← shape 固定
```

**优点**：实现简单、shape 整齐、几乎所有 attention 实现都支持
**缺点**：PAD token 也参与计算，**显存和算力浪费**（短样本多的时候浪费很大）

### THD（packed，变长拼接）

```
Time × Heads × Dim   （把 batch 维"压扁"了）
```

把所有样本的有效 token **首尾相连拼成一个长序列**，记录每条样本的边界：

```
样本1: [t1 t2 t3]
样本2: [t1 t2 t3 t4 t5]
样本3: [t1]
        ↓
Tensor: [9, hidden, dim]
cu_seqlens: [0, 3, 8, 9]   ← 累计序列长，标记每条样本起止
```

**优点**：**没有 PAD，零浪费**；可以按 token 总数装箱（=动态 batch 的本质）
**缺点**：attention 算子必须支持 `cu_seqlens` 形式的 varlen API（FlashAttention varlen、xformers block_diagonal mask）；线性注意力 / Mamba / GDN 这种"状态机"型的 op，做 varlen 比较复杂

## 二、GDN（Gated Delta Net）为什么不支持 THD

GDN 是 Qwen3.5 用的**线性注意力变体**（Yang et al., 2024，FLA 库实现）。它和 softmax attention 在数学上完全不同：

- **Softmax attention**：QKᵀ → softmax → V，每个 token **独立**算注意力分数；varlen 只需要在 mask 里区分样本边界，FlashAttention 的 cu_seqlens 接口能优雅处理
- **GDN 线性注意力**：内部维护一个 **状态矩阵 S** 沿 token 维**递推更新**（类 RNN 的 chunkwise 算法）：
  ```
  S_t = α_t · S_{t-1} + k_t v_tᵀ
  o_t = q_tᵀ S_t
  ```
  状态 S 在样本之间**不能串**（样本 2 的状态被样本 1 的尾部污染就完全错了），但 chunked 实现是**按等长 chunk 切分整个长序列**做的，varlen 边界穿插到 chunk 内部时，状态重置逻辑非常复杂

Megatron-LM 的 GDN 算子（截至 2026.04 版本）**只实现了 BSHD 路径**，没有 cu_seqlens-aware 的 varlen 版本。THD 输入进去会算错，所以必须强制 BSHD。

## 三、三个参数的依赖关系

```
GDN 不支持 THD
    ↓
必须走 BSHD 计算路径
    ↓
不能"去 padding"（remove_padding 就是 BSHD → THD）
    ↓
model.use_remove_padding = False
actor.megatron.use_remove_padding = False
    ↓
没法按 token 数装箱（dynamic batching 依赖 THD packing）
    ↓
actor.use_dynamic_bsz = False   ← 只能按"条数"做静态 batch
```

### 三个参数各自的作用

**1. `actor_rollout_ref.model.use_remove_padding`**（顶层模型选项）
- 全局指示这个模型是否能消化 packed 输入
- 用于 model wrapper / data collator 的预处理决策
- Qwen3.5 GDN 不行 → False

**2. `actor_rollout_ref.actor.megatron.use_remove_padding`**（Megatron 后端专用）
- 强制 Megatron 的 forward 走 BSHD 路径，不调 varlen kernel
- 即使 attention backend（如 flash_attention_2）本身支持 varlen，也压成 BSHD
- Qwen3.5 GDN 不行 → False

**3. `actor_rollout_ref.actor.use_dynamic_bsz`**（数据装箱策略）
- True：按 `ppo_max_token_len_per_gpu` token 数装箱（依赖 THD 喂给 model）
- False：按固定 `ppo_micro_batch_size_per_gpu` 条数喂（BSHD 兼容）
- 上面两个 remove_padding 都 False 时，dynamic_bsz 必须 False，否则 collator 会报错（混用 BSHD 模型 + THD 数据）

## 四、对你 35B-A3B 训练的实际影响

### 性能损失
对一个 prompt 8192 + response 16384 = **24576 token** 的样本：
- 短样本（响应只有 1000 token）实际有效长度 = 8192 + 1000 = 9192
- BSHD 模式下还是按 24576 算，**浪费 (24576-9192)/24576 ≈ 63% 的算力**
- 如果用 THD，能省下这 63% → 训练更快、显存更松

所以 GDN 模型在当前阶段训练就是**比 softmax-attention 模型贵**，这是架构带来的固有 cost。

### 调参影响
既然 dynamic_bsz 用不了，micro_bs_per_gpu 就要按"最长样本能不能塞下"来定：
```bash
actor_rollout_ref.actor.ppo_micro_batch_size_per_gpu=1   # 长样本下，1 也可能 OOM
actor_rollout_ref.rollout.log_prob_micro_batch_size_per_gpu=1
actor_rollout_ref.ref.log_prob_micro_batch_size_per_gpu=1
```
即使你的 batch 里平均长度只有 5000，但因为有几条 24000 的，micro_bs 还是只能 1。**全 batch 被最长那条拖累**。

### 数据预处理对策
既然没法动态装箱，**控制数据分布**就更重要：
- `data.filter_overlong_prompts=True` 已经过滤了超长 prompt
- 可以考虑按长度分桶训练（先短后长 / 按长度排序成 batch）减少 padding 浪费
- `data.truncation='error'` 严格报错而不是默默截断，避免脏数据

## 五、什么时候这个限制会解除？

官方注释明说：
> Once Megatron-LM adds THD support for Qwen3.5 GDN, `use_remove_padding` can be set to True for better performance.

需要看：
1. Megatron-LM 上游给 Mamba/GDN 类算子加 varlen 支持（PR 已经在做）
2. 或者用替代实现（FLA 库的 chunked GDN 支持 cu_seqlens 但还没集成进 Megatron）

短期内：跟着官方推荐配置（False, False, False），等 verl 更新示例时再切。

## 六、面试要点

- **THD vs BSHD 是张量布局术语**：T=拼接的总 token 数 vs B=batch 维度 + 显式 pad
- THD 装包用于零浪费计算，但需要算子支持 `cu_seqlens` (FlashAttention varlen)
- **线性注意力 / 状态机型 op (GDN/Mamba) 实现 varlen 比 softmax attention 困难得多**，因为状态会在样本之间串
- Qwen3.5 GDN 当前 Megatron 实现只有 BSHD，**所以 use_remove_padding 和 use_dynamic_bsz 必须 False**，是被架构限制锁死，不是性能选择
- 后果：长 batch 下显存浪费大、micro_bs 被最长样本拖累；短期内只能靠数据侧（过滤超长、按长度分桶）缓解
- 类似限制不只 Qwen3.5：早期 Mamba、RWKV 进 Megatron 时都遇到过同样的过渡期

# 批大小整除性陷阱：为什么 48 卡下 train_batch_size=512 跑不起来

短答：**`train_batch_size × rollout.n` 必须能被 `dp × micro_per_gpu` 整除**，且 **`ppo_mini_batch_size × rollout.n` 也必须能被 `dp × micro_per_gpu` 整除**。GPU 数量变化（比如从 16 卡扩到 48 卡），DP 组数随之变化，原本能整除的 batch 大小可能突然不行了。

## 报错长这样

```
AssertionError: real_train_batch_size (4096) must be divisible by minimal possible batch size (24)
```

源码位置：[verl/utils/config.py:108-113](../../verl/utils/config.py#L108)

```python
megatron_dp = n_gpus // (model_parallel_size * context_parallel_size)
minimal_bsz = megatron_dp * config.actor_rollout_ref.actor.ppo_micro_batch_size_per_gpu

real_train_batch_size = data.train_batch_size * rollout.n
assert real_train_batch_size % minimal_bsz == 0
```

## 三条整除约束完整版

verl 训练能起来，需要**同时满足**：

| 约束 | 数学表达 | 含义 |
|---|---|---|
| ① 全局批 | `train_batch × rollout.n % minimal_bsz == 0` | 全 step sample 总数能均匀分到所有 DP 组 |
| ② mini-batch | `mini_batch × rollout.n % minimal_bsz == 0` | 每次 optimizer.step 的 sample 能均匀分 |
| ③ batch 层级 | `train_batch % mini_batch == 0` | mini-batch 能整数次切分 train-batch |

其中 `minimal_bsz = (n_gpus / (TP × PP × CP)) × ppo_micro_batch_size_per_gpu`。

## 16 卡 vs 48 卡 minimal_bsz 对比

固定 TP=2, PP=1, CP=1, micro=1：

| n_gpus | DP 组数 | minimal_bsz | 质因子分解 | "好整除"的批大小 |
|---|---|---|---|---|
| 8（1 节点） | 4 | 4 | 2² | 任意 2 的倍数 |
| 16（2 节点） | 8 | 8 | 2³ | 任意 8 的倍数 |
| **48（6 节点）** | **24** | **24** | **2³ × 3** | **必须是 3 的倍数** |
| 64（8 节点） | 32 | 32 | 2⁵ | 任意 32 的倍数 |

⚠️ **关键观察**：48 卡的 minimal_bsz 含**质因子 3**。而 RL 常用的批大小（128/256/512/1024）都是 **2 的幂**，**永远不被 3 整除**。所以从 16 卡迁到 48 卡，旧 batch 配置必然失效。

## 解决思路

需要让 `train_batch_size` 和 `ppo_mini_batch_size` **都是 3 的倍数**（因为 `rollout.n=8` 是 2 的幂，无法消掉 3）。

### 48 卡推荐配置（保持 4 次更新/step）

| | 选项 A（保守 ≈ 原 batch）| 选项 B（充分利用 48 卡）|
|---|---|---|
| `train_batch_size` | 384 | 576 |
| `ppo_mini_batch_size` | 96 | 144 |
| `micro_per_gpu` | 1 | 1 |
| 总 sample/step | 3072 | 4608 |
| mini-batch sample | 768 | 1152 |
| 每 mini-batch 累积 | 32 | 48 |
| 每 step 更新次数 | 4 | 4 |
| 校验 ① | 3072 % 24 = 0 ✓ | 4608 % 24 = 0 ✓ |
| 校验 ② | 768 % 24 = 0 ✓ | 1152 % 24 = 0 ✓ |
| 校验 ③ | 384 / 96 = 4 ✓ | 576 / 144 = 4 ✓ |

### 不同 TP 组合的可行性

48 卡下试图换 TP 让 minimal_bsz 变成 2 的幂：

| TP | DP 组数 | minimal_bsz | 备注 |
|---|---|---|---|
| TP=2 | 24 | 24 | **2³ × 3**，需要 3 的倍数 batch |
| TP=3 | 16 | 16 | 2⁴，2 的幂友好；但 **TP=3 要求 hidden 是 3 的倍数**，多数模型不支持 |
| TP=4 | 12 | 12 | 2² × 3，照样含 3 |
| TP=6 | 8 | 8 | 2³ 友好，但 TP=6 通信开销大 |
| TP=8 | 6 | 6 | 2 × 3，含 3 |

→ **48 卡场景没有"既保 TP=2 又消掉质因子 3"的捷径**，必须改 batch。

## 调参公式（背下来）

```
要让某节点数 N 下的 RL 跑通，需要：
    dp = N × n_gpus_per_node / (TP × PP × CP)
    minimal_bsz = dp × micro_per_gpu

    train_batch_size  必须是  (minimal_bsz / rollout.n) 的倍数（向上取整考虑余数）
    ppo_mini_batch_size 必须是 (minimal_bsz / rollout.n) 的倍数
    train_batch_size 必须能被 ppo_mini_batch_size 整除（最好整数倍）
```

更直接的口诀：**把 `minimal_bsz / gcd(minimal_bsz, rollout.n)` 算出来，它是 `train_batch_size` 和 `ppo_mini_batch_size` 必须是的"最小步长"**。

举例：
- 16 卡：minimal_bsz=8, rollout.n=8, gcd=8, 步长=1 → batch 随便配
- 48 卡：minimal_bsz=24, rollout.n=8, gcd=8, 步长=3 → batch **必须是 3 的倍数**
- 64 卡：minimal_bsz=32, rollout.n=8, gcd=8, 步长=4 → batch 必须是 4 的倍数（但 4 是 2 的幂，常用配置都满足）

## 面试要点

- RL 训练换 GPU 数量时**最容易翻车的不是显存，而是整除性约束**：minimal_bsz 跟 GPU 数走，含质因子 3 / 5 / 7 时所有 2 的幂 batch 都失效
- **48 卡是个典型陷阱**：6 节点 × 8 卡看起来挺规整，但 6 = 2 × 3，DP 组数带进质因子 3，batch 必须配 3 的倍数（384、480、576...）
- 不能通过减小 rollout.n 解决：rollout.n=8 没法把质因子 3 吃掉
- 不能通过加 TP 解决（48 卡下 TP=2/4/8 都引入或保留 3，TP=3 多数模型不支持）
- 工程经验：**RL 训练改节点数时，一并审视 `train_batch_size / ppo_mini_batch_size / minimal_bsz` 的整除关系**，不要只看显存够不够

## 面试金句

> 我从 16 卡扩到 48 卡时踩过一个坑：报错说 `real_train_batch_size (4096) must be divisible by minimal_bsz (24)`。原因是 48 卡 / TP=2 → DP 组数 24，minimal_bsz=24 含质因子 3，而 RL 常用的 2 的幂 batch（128/256/512）都不被 3 整除。解决方法是把 `train_batch_size` 和 `ppo_mini_batch_size` 都改成 3 的倍数（如 384/96）。这件事让我意识到 **RL 训练换节点数不只是看显存，还要查 batch 的整除性，DP 组数携带的质因子会传染到所有 batch 参数**。
