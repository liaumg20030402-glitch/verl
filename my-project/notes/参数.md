## 三层 batch 的关系

GRPO 训练一个 step 的数据流,**从粗到细有三层**:

```
┌─────────────────────────────────────────────────┐
│ train_batch_size = 128 (prompt 数)             │  ← Rollout 阶段
│ × rollout.n = 8 (每 prompt 采 8 条 response)   │
│ = 1024 条 (prompt, response) 样本              │
└─────────────────────────────────────────────────┘
                    ↓
        reward 打分,算 advantage
                    ↓
┌─────────────────────────────────────────────────┐
│ ppo_mini_batch_size = 32                        │  ← 训练阶段
│ 把 1024 条样本切成 1024/32 = 32 个 mini-batch   │
│ 每个 mini-batch 做一次参数更新(optimizer.step)│
└─────────────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────────────┐
│ ppo_micro_batch_size_per_gpu = 1                │  ← 前向/反向
│ 32 条 mini-batch 分摊到 8 张卡,每卡 4 条      │
│ 每卡每次只前向 1 条(显存约束),4 次梯度累加 │
└─────────────────────────────────────────────────┘
```

## 各层的具体作用

### 1. `train_batch_size`:定义"一个 RL step"的规模

**意思:每一个训练 step,从数据集拿多少个 prompt 出来做 rollout。**

- 你的值:128
- 这 128 个 prompt 会一次性送给 vLLM 做 rollout
- 每个 prompt 采 `rollout.n=8` 条 response
- 产出 128 × 8 = 1024 条训练样本

**这是"RL 数据生产"的单位**,决定每次 reward 估计的 variance、训练的探索多样性。

### 2. `ppo_mini_batch_size`:定义"一次梯度更新"的规模

**意思:拿到这 1024 条样本后,每次用多少条做一次 `optimizer.step()`。**

- 你的值:32
- 1024 / 32 = **32 次梯度更新**(一个 step 内!)
- 每 32 条样本算 loss、反向、更新一次权重

**这是"监督学习里我们熟悉的 batch size"**,决定每次参数更新的 noise 和 step size。

### 3. `ppo_micro_batch_size_per_gpu`:定义"一次前向"的规模

**意思:每张 GPU 每次前向传播处理多少条样本。**

- 你的值:1
- 和 `ppo_mini_batch_size` 的关系:`mini_batch_size / (总 GPU 数 × micro_bs_per_gpu) = 累积次数`
- 你 8 卡,32 / (8 × 1) = **每张卡梯度累积 4 次**,再统一 optimizer.step

**这是纯粹的显存适配参数**,和训练效果无关,只影响训不训得动。

## 关键区别:监督学习 vs PPO

在普通的 SFT 里,"batch size" 只有一个值,既是数据规模也是梯度更新规模。但 PPO/GRPO 有一个**"on-policy 数据要被反复训练"**的特性:

```
拿一批 rollout 数据 → 训 N 次 → 重新 rollout
```

**这就是为什么要拆成 train_batch 和 mini_batch**:

- `train_batch=128`:每次 rollout 生成 128 prompt 的数据
- `ppo_mini_batch=32`:这批数据被切成 32 份,每份训一次(总共 32 次更新)
- 然后才重新 rollout 下一批

换句话说,**一次 rollout 的数据被反复用了 32 次更新**。这就是 PPO 相对于"纯 online RL"的数据效率来源。

### 还有第四个参数:`ppo_epochs`

虽然你没显式配(verl 默认 1),但概念上还有一个:

```
ppo_epochs = 2 意味着 1024 条样本会被整体训 2 遍
→ 总共 1024 × 2 / 32 = 64 次 optimizer.step
```

## 用 PyTorch 伪代码理解

```python
# 一个 GRPO step 的内循环
for step in range(total_steps):
    # === Rollout 阶段 ===
    prompts = dataloader.next(batch_size=128)              # train_batch_size
    responses = vllm.generate(prompts, n=8)                # rollout.n
    rewards = reward_fn(responses)                         # 1024 条
    advantages = compute_grpo_advantage(rewards)           # 组内归一化
    
    # === 训练阶段 ===
    all_samples = list(zip(prompts, responses, advantages))
    
    for epoch in range(ppo_epochs):                        # 默认 1
        shuffle(all_samples)
        
        for mini_batch in chunks(all_samples, size=32):    # ppo_mini_batch_size
            # 梯度累积(显存约束)
            for micro_batch in chunks(mini_batch, size=1*8):  # micro × n_gpus
                loss = compute_ppo_loss(micro_batch)
                loss.backward()
            
            optimizer.step()                               # 每个 mini_batch 更新一次
            optimizer.zero_grad()
```

## 几个常见困惑

### Q1:train_batch 和 mini_batch 的比值代表什么?

`train_batch_size × rollout.n / ppo_mini_batch_size = 每步 optimizer.step 次数`

你的配置:`128 × 8 / 32 = 32 次更新`。

- **比值大**(你现在 32 倍):每批 rollout 数据反复训很多次 → 数据效率高,但可能过拟合当前 batch,策略偏离 rollout 时的策略过多(off-policy bias)
- **比值小**(比如 4 倍):数据用完就扔 → 数据效率低,但策略始终接近 on-policy,训练稳定

**经验值**:PPO 通常 4-16 次更新是安全区,你的 32 次**偏激进**,早期训练可能不稳。建议:

```bash
actor_rollout_ref.actor.ppo_mini_batch_size=64   # 32 → 64,降到 16 次更新
```

### Q2:为什么 mini_batch 不直接等于 train_batch?

如果 `ppo_mini_batch_size = train_batch_size × rollout.n = 1024`,那一个 step 只更新 1 次参数。这样:
- 好处:100% on-policy,最稳定
- 坏处:数据用一次就扔,**采样极贵的 rollout 白费**(你跑 35B MoE 的 rollout 很贵!)

**折中**:拆成多个 mini_batch,用 PPO clip 约束策略不飘太远,同一批数据复用多次。

### Q3:micro_batch 怎么调?

- OOM 了 → 降 `ppo_micro_batch_size_per_gpu`
- 显存还有余 → 提高它,减少梯度累积次数,训练更快
- **只影响显存和速度,不影响效果**(梯度累积在数学上等价于大 batch)

你现在 `=1`,在 H200 141G 上**太保守**,可以试 `=2` 或 `=4`,提速 2-4 倍。

## 一、核心业务指标:reward / score / acc

这一组直接反映"模型到底做得好不好",是你最该盯的。

### `critic/rewards/mean`(也叫 `reward/mean`)

**一个 rollout batch 内所有样本的平均 reward。**

对你的 blzk 规则奖励来说,reward 只有 0.0 和 1.0 两个值,所以 `mean` 实际上就是"正确率"。

**健康曲线:**
- **初始值**:取决于 base model 能力。Qwen3.5-35B-A3B 这种强模型,医病历质控任务**冷启动可能就有 0.3~0.6**(模型已经能输出合格 JSON,部分结论对)
- **上升斜率**:前 50-200 step 应该能看到明显上升,之后趋于平缓
- **最终值**:0.7~0.95 之间,很少能到 1.0(数据里总有模糊边界样本)

**异常情况:**

| 现象 | 说明 |
|------|------|
| 一直是 0 | reward function 有 bug,或者模型输出格式完全对不上期望 |
| 卡在某个低值不动(比如 0.15) | 格式分太严,模型学会了某个"局部最优"(比如全答合格) |
| 先上升后崩盘 | **reward hacking** 或 entropy collapse,典型 failure mode |
| 上下剧烈震荡 | learning rate 太大 / batch 太小 / KL coef 太小 |

### `critic/score/mean`

**这个是原始打分**(reward function 返回的 `score`),在某些 reward manager 里等同于 reward,**在 dapo 里可能略有不同**。

dapo manager 会做一个特殊处理:**对所有样本 reward 都相同的 group 过滤掉**(避免 entropy 塌缩)。所以:

- `score/mean` = 所有样本的原始分数均值
- `reward/mean` = 被保留用来训练的样本的分数均值(dapo 过滤后)

如果 `reward/mean` 明显高于 `score/mean`,说明 dapo 过滤了很多"全对"或"全错"的 group。**dapo 保留比例低于 30%** 就要警觉了——说明组内 reward 多样性不够。

### `acc`(accuracy,你 reward function 返回的字段)

你的 reward 函数返回了 `"acc": bool(score > 0.5)`,这会被 dapo 聚合成 `critic/acc/mean`。

对你的场景,**acc ≈ reward**,因为是 0/1 打分。但如果你后续引入格式分(比如 0.3 格式正确 + 0.7 答案正确),acc 和 reward 就会分离——**acc 看的是最终答案是否对,reward 看的是总分**。这时 acc 比 reward 更能反映"任务完成度"。

### `judge_reason` 分布(你的自定义字段)

这是你 reward function 里返回的字段,dapo 会统计每种 reason 的占比。健康状态:

```
ok:                    60-90% (训练中期应该占大多数)
mismatch:              10-30% (答错但格式对)
json_not_found:        0-5%   (格式错误,应该随训练下降)
key_mode_not_matched:  0-5%
json_parse_failed:     <1%
```

**异常**:如果 `json_not_found` 或 `key_mode_not_matched` 训练中不下降甚至上升,说明**模型没在学格式**,要检查 reward 是不是没区分度(全零导致没梯度)。

## 二、优势函数指标:advantage

### `critic/advantages/mean` 和 `critic/advantages/std`

**Advantage 反映"这条 response 比同组里的平均水平好多少"**。GRPO 里,advantage = (reward - group_mean) / group_std,所以理论上:

- **mean 应该约等于 0**(因为是组内中心化)
- **std 应该约等于 1**(因为归一化了)

**健康状态:**
- `advantages/mean` ∈ [-0.1, 0.1](非常接近 0)
- `advantages/std` ∈ [0.7, 1.3]

**异常:**

| 现象 | 说明 |
|------|------|
| `std` 持续很小(< 0.3) | 组内 reward 高度一致——要么模型全对了,要么全错了。dapo 会过滤,但如果大量组都这样,有效训练信号很少 |
| `std` 持续很大(> 2) | reward 方差爆炸,可能 reward function 返回了异常值(NaN、inf) |
| `mean` 显著偏离 0 | 归一化有问题,检查 GRPO 实现(一般是框架 bug,不是你的问题) |

### `critic/advantages/max` 和 `critic/advantages/min`

这两个对 **detect outlier** 很有用。健康范围:

- `max` ∈ [1.5, 3.0]
- `min` ∈ [-3.0, -1.5]

如果看到 `max=50` 这种值,肯定是某个样本 reward 异常飙高——去 rollout_data_dir 里找那条样本看看。

## 三、策略更新指标:loss 和 KL

### `actor/pg_loss`(policy gradient loss)

**PPO 的策略梯度损失**。这是优化的目标,但它的数值变化**反直觉**:

- **健康状态**:负数,绝对值很小(-0.01 ~ -0.1)
- **不应期待它单调下降**!pg_loss 上下波动是正常的,因为它反映的是"当前 policy 和 rollout policy 的差异 × advantage"
- 重点看它**有没有突变成 NaN/inf**——那是训崩的信号

### `actor/kl_loss` 和 `actor/kl_coef`

KL 散度:当前 policy 和 reference model(base model)的差距。

你的配置:
```bash
actor_rollout_ref.actor.kl_loss_coef=0.01
actor_rollout_ref.actor.kl_loss_type=low_var_kl
```

**健康状态:**
- `kl_loss` 初始接近 0,训练中**缓慢上升**
- 到训练后期稳定在 **0.1 ~ 2.0** 之间
- 绝对不应该爆炸到 10+(说明 policy 跑飞了)

**异常:**

| 现象 | 原因 | 修复 |
|------|------|------|
| KL 飙到 10+ | policy 偏离 base 太远,reward hacking | 调大 `kl_loss_coef`(0.01 → 0.05) |
| KL 一直 ≈ 0 | policy 没更新,reward 没梯度 | 检查 reward 是否全零 |
| KL 震荡剧烈 | batch 太小或 lr 太大 | 增大 batch 或降 lr |

### `actor/entropy_loss` 或 `actor/entropy`

策略的熵。**衡量模型输出的多样性**。

你的配置 `entropy_coeff=0`,所以 entropy 不直接进 loss,但还是会被记录。

**健康状态:**
- **初期**高(1.5 ~ 3.0),模型输出多样
- **中期**缓慢下降(0.8 ~ 1.5),模型开始收敛
- **后期**稳定在 0.3 ~ 1.0

**异常(最危险):entropy collapse**
- 如果 entropy **快速掉到 < 0.1**,模型开始对所有 prompt 输出几乎一样的内容——**rollout 采样 8 条全部一样**,失去 RL 训练信号
- 这是 GRPO 最常见的 failure mode,**dapo reward manager 就是为了防止这个**(过滤全同 group)
- 如果遇到,把 `entropy_coeff` 设为正值(比如 0.001)加回来

### SFT(监督学习):确实只有一种前向

```

    ↓ optimizer.step()
```

SFT 里只有一次"前向",label 是人工给的,loss 就是预测 vs 真值的差距。**简单、直接**。

### RL(GRPO/PPO):有多个前向,而且分工不同

```
prompt batch
    ↓ 【前向 #1: Rollout 生成】—— vLLM 里多次自回归前向
    ↓ 得到 responses
    ↓ 【Reward 阶段】—— 给 responses 打分(不是 loss)
    ↓ 得到 rewards → 算 advantages
    ↓ 【前向 #2: 计算 log_prob】—— actor 模型对已生成的 response 做一次前向
    ↓ 【前向 #3: 计算 ref log_prob】—— ref 模型做一次前向(算 KL)
    ↓ 【计算 loss】—— 用 log_prob、advantage、ref_log_prob 组合成 PPO loss
    ↓ 反向传播
    ↓ optimizer.step()
```

## 一、什么是 Rollout

**Rollout ≠ 单次前向传播。Rollout 是"让模型生成一整条 response"的过程。**

### Rollout 内部发生了什么

假设 prompt 是 "1+1=",模型要生成 "2<eos>",一共 2 个 token。**生成每个 token 都要一次前向传播**:

```
t=0: 输入 "1+1="           → 前向 → 输出 token 概率分布 → 采样出 "2"
t=1: 输入 "1+1=2"          → 前向 → 输出 token 概率分布 → 采样出 "<eos>"
                                                       → 停止
```

**所以生成一条 2 个 token 的 response = 做 2 次前向传播**。

你的配置:`max_response_length=2048`,意味着极端情况下一条 response 要做 **2048 次自回归前向**。每次前向生成 1 个 token(vLLM 有 paged attention、KV cache、batching 等优化,但逻辑上就是一次次前向)。

再乘以你的 batch:
- `train_batch_size=128` 个 prompt
- `rollout.n=8` 条 response/prompt
- **1024 条 response × 平均 500 token = 50 万次前向传播**

**这就是为什么 rollout 阶段这么慢、这么贵**。vLLM 存在的意义就是让这一步高效。

### Rollout 的特点

1. **不计算梯度**(`torch.no_grad()`),也不算 loss
2. **纯推理**,目的只是"生成数据"
3. **用 vLLM 或类似推理引擎**,因为要极致优化 KV cache、batching
4. 生成完就把 responses 交给下游处理

### Rollout 和"计算 log_prob 的前向"不是同一个

注意一个微妙区别:
- **Rollout 时的前向**:每次只算下一个 token 的概率,采样出来,丢掉其他 logits
- **训练时的前向(算 log_prob)**:对已经生成好的完整 response,一次性前向,拿到**每个位置的 log_prob**,用来算 loss

为什么要再前向一次?因为 rollout 时用的可能是 vLLM 的引擎(独立进程、BF16 推理),而训练时的 actor model 权重**正在被更新**,两者是分离的。所以训练时要用当前 actor 的权重**重新前向一次**算准确的 log_prob。

## 二、什么是 Reward 和 Advantage

**这俩都不是 loss,而是"监督信号"。**

### Reward

Reward 是**外部给的分数**,告诉模型"这条 response 好不好"。来源可以是:
- 规则函数(你的 blzk:JSON 格式对 + 结论对 = 1,否则 = 0)
- 奖励模型(用另一个 LLM 打分)
- 人工标注

**Reward 不可微**,它是一个黑盒打分函数——你的 `compute_score_blzk_rule` 里又是正则又是 JSON 解析,根本无法反向传播。**Reward 不参与梯度计算**。

### Advantage

Advantage = reward 的**相对化**表示。

在 GRPO 里:
```
advantage_i = (reward_i - group_mean) / group_std
```

含义:"这条 response 在它所属的 group(同一个 prompt 的 8 条 response)里,比平均水平好多少个标准差"。

**为什么不用 reward 直接算 loss?**

- Reward 的绝对值可能系统性偏高或偏低(比如任务很难,reward 普遍低)
- 用 advantage 消除这种偏差,让训练更稳定
- Advantage > 0 的样本:要"鼓励"这种行为
- Advantage < 0 的样本:要"抑制"这种行为

### Reward 和 Advantage 都只是"数字标签"

它们的作用类似 SFT 里的 label —— **告诉模型"这条数据的梯度方向应该是正还是负,强度多大"**。但本身不是 loss。

## 三、真正的 Loss:PPO Policy Gradient Loss

**Loss 是根据 log_prob 和 advantage 组合出来的。**

### 简化版的 PPO loss

```python
# 对于每条 (prompt, response, advantage) 样本
log_prob_new = actor_model(prompt, response)      # 当前 policy 的 log_prob
log_prob_old = rollout_log_prob                   # rollout 时 policy 的 log_prob(已保存)

ratio = exp(log_prob_new - log_prob_old)          # 重要性采样比率

# PPO clip loss
loss_unclipped = -ratio * advantage
loss_clipped   = -clip(ratio, 1-ε, 1+ε) * advantage
pg_loss = max(loss_unclipped, loss_clipped).mean()

# KL 正则
log_prob_ref = ref_model(prompt, response)
kl = log_prob_new - log_prob_ref
kl_loss = kl.mean()

# 总 loss
total_loss = pg_loss + kl_coef * kl_loss
total_loss.backward()
```

**关键点:**

1. **loss 是 `log_prob × advantage` 的组合**——log_prob 可微,advantage 是常数
2. 梯度通过 `log_prob_new` 回传到 actor 参数
3. Advantage 只是个"乘数",告诉梯度"往哪个方向走,走多大步"

### 用 SFT 对比类比

| | SFT | PPO/GRPO |
|---|-----|-----------|
| 监督信号 | label(目标 token) | advantage(标量) |
| 模型输出 | logits 上每个 token 的概率 | log_prob(已生成 token 的概率) |
| Loss | -log P(label)(交叉熵) | -log P(response) × advantage |
| 信号来源 | 人工标注的正确答案 | reward function 打分 + 组内归一化 |

可以粗略地看成:**PPO 是"带权重的 SFT"**——advantage 是每条样本的权重,rollout 生成的 response 是"伪 label",按 advantage 的正负和大小来决定学习方向和强度。

## 四、对应到 verl 的实际流程

帮你把整个链路再串一遍,用你的配置数字:

```
┌─ 一个 Training Step ─────────────────────────────────────┐
│                                                           │
│ Step 1: 从 parquet 拿 128 个 prompt                      │
│                                                           │
│ Step 2: Rollout(vLLM 上推理,不训练)                   │
│   - 每个 prompt 采 8 条 response (rollout.n=8)          │
│   - 每条 response 最多 2048 个 token                     │
│   - 内部: 约 50 万次 autoregressive 前向传播           │
│   - 产出: 1024 条 (prompt, response) + 每 token 的      │
│           log_prob_old                                    │
│                                                           │
│ Step 3: Reward 打分(你的 reward function)              │
│   - 对每条 response 算 score (0.0 或 1.0)                │
│   - 【不做梯度计算】                                     │
│                                                           │
│ Step 4: 算 Advantage(GRPO 归一化)                      │
│   - 每 8 条同 prompt 的 response 为一组                  │
│   - advantage = (reward - group_mean) / group_std        │
│                                                           │
│ Step 5: 前向算 log_prob                                  │
│   - Actor model 前向一次 → log_prob_new                  │
│   - Ref model 前向一次 → log_prob_ref                    │
│   - 【这里的前向是一次性整句过,不是自回归】            │
│                                                           │
│ Step 6: 算 Loss 并反向                                   │
│   - pg_loss = -ratio × advantage (带 clip)              │
│   - kl_loss = log_prob_new - log_prob_ref                │
│   - total = pg_loss + 0.01 × kl_loss                     │
│   - backward + optimizer.step()                          │
│                                                           │
│   - 按 ppo_mini_batch_size=32 分批,重复 step 5-6 共 32  │
│     次                                                    │
│                                                           │
└───────────────────────────────────────────────────────────┘
         ↓ 进入下一个 step,重新 rollout
```

## 五、一个常见的误解

> "既然 reward 不可微,那梯度怎么回传到 model 上?"

**关键 trick:梯度不从 reward 回传,而是从 `log_prob_new` 回传。**

```python
loss = -log_prob_new × advantage   # advantage 是常数(detach 过)
```

反向传播时:
- `advantage` 被视为常数(`.detach()` 处理过)
- `log_prob_new` 是可微的(actor 前向算出来的)
- 梯度通过 `log_prob_new` 传到 actor 的参数

这就像**"用当前模型生成的 token 当伪标签,但给每条样本加了 advantage 的权重"**——如果 advantage > 0(这条比平均好),就鼓励模型更愿意输出这种 token;如果 < 0,就抑制。

## 简洁回答你的原问题

| 你的理解 | 实际是 |
|---------|--------|
| LLM 的 rollout 就是前向传播 | ❌ Rollout 是"用模型生成完整 response"的过程,**包含几百到几千次自回归前向**,而且和训练时的前向是分开的(vLLM vs actor) |
| 计算 loss 就是计算 reward 和 advantage | ❌ Reward 和 advantage 是**监督信号(标签)**,不是 loss。真正的 loss 是 `-log_prob × advantage`(PPO clip + KL 正则),计算过程只有 **log_prob** 参与梯度,reward 和 advantage 只是**数值权重** |
